# uni-app x 项目骨架与基础页面搭建 Spec

## Why
当前仓库仅包含 uni-app x 初始化默认页，尚未形成可持续扩展的工程结构，也无法承接后续业务页面开发。需要先建立统一的项目骨架、页面路由与基础组件能力，并按 `H5` 目录中的参考稿落地首批静态页面。

## What Changes
- 搭建面向 uni-app x 的推荐目录结构，明确页面、组件、工具、状态和静态资源的组织方式
- 默认引入并接入 `uni-ui x` 组件库，作为基础 UI 能力来源
- 重构应用入口与页面注册方式，建立清晰的页面路由和页面分层
- 基于 `H5` 参考稿实现首批静态页面和页面跳转，不接入真实接口
- 沉淀首批可复用的布局、卡片、列表项、操作区等基础业务组件
- 补充统一的视觉基础配置，包括全局样式、主题色、圆角、间距和页面容器约定

## Impact
- Affected specs: 新增 `app-shell`、`page-routing`、`ui-foundation`、`health-demo-pages` 能力规格
- Affected code: `App.uvue`、`main.uts`、`pages.json`、`manifest.json`、`uni.scss`、`pages/**`、`components/**`、`static/**`

## ADDED Requirements
### Requirement: uni-app x 工程骨架
系统 SHALL 提供适用于 uni-app x 项目的基础工程结构，能够支撑页面扩展、公共组件复用和后续业务接入。

#### Scenario: 初始化项目骨架
- **WHEN** 开发者进入项目根目录查看代码结构
- **THEN** 可以看到按职责划分的页面目录、公共组件目录、静态资源目录和基础工具目录
- **AND** 目录命名与页面注册路径保持一致，避免后续页面扩展时混乱

### Requirement: uni-ui x 基础能力接入
系统 SHALL 默认引入 `uni-ui x`，并在页面搭建中优先复用其可用基础组件能力。

#### Scenario: 页面使用基础 UI 组件
- **WHEN** 开发者实现列表、按钮、输入区、弹层或导航辅助区域
- **THEN** 优先复用 `uni-ui x` 兼容组件或其设计模式
- **AND** 不重复封装已可直接满足需求的基础控件

### Requirement: 首批页面路由与结构
系统 SHALL 提供与参考稿对应的首批页面，并完成页面注册与可达跳转。

#### Scenario: 页面注册完成
- **WHEN** 应用启动并进入首页
- **THEN** 首页可以正常渲染为对话场景页
- **AND** 用户可以通过页面操作进入历史对话、健康档案、系统设置、账号设置、举报反馈和关于我们等页面

#### Scenario: 页面结构对齐参考稿
- **WHEN** 开发者查看已实现页面
- **THEN** 页面信息架构应与 `H5` 目录参考稿保持一致
- **AND** 页面可先使用静态文案和占位数据完成布局表达

### Requirement: 统一页面容器与视觉规则
系统 SHALL 为首批页面提供统一的页面容器、卡片样式、间距规则和背景表达，降低后续页面实现成本。

#### Scenario: 新页面复用视觉基础
- **WHEN** 新增一个同风格业务页面
- **THEN** 可以直接复用统一的页面容器、标题区、内容卡片和底部操作区样式约定
- **AND** 不需要在每个页面重复定义相同的背景色、圆角和间距值

### Requirement: 静态演示数据支持
系统 SHALL 为首批页面提供最小静态演示数据，使页面在无接口接入时也能完整展示结构。

#### Scenario: 无接口环境预览页面
- **WHEN** 项目在本地直接运行
- **THEN** 各页面可以通过本地静态数据渲染标题、说明文案、列表项和按钮态
- **AND** 不依赖后端服务即可完成页面预览和联调准备
