# uni-ui x 手动导入桥接方案设计

## 背景

当前仓库是标准 `uni-app x` 空工程，根目录不存在 `package.json`，也不存在 `uni_modules`。根据 DCloud 官方文档，`uni-ui x` 是面向 `uni-app x` 的扩展组件库，推荐通过插件市场导入到项目内的 `uni_modules`，而不是在当前工程形态下直接依赖 npm 包。

本次工作采用方案 C：后续由开发者通过 HBuilderX 手动导入真实 `uni-ui x` 组件；当前仓库内先完成衔接层、目录规范和占位结构，为后续真实导入留出稳定接入口，但不伪造任何不存在的真实组件包。

## 目标

- 明确项目内 `uni-ui x` 的推荐接入方式与边界
- 创建与后续真实导入兼容的目录和文档结构
- 提供一层仓库内可维护的业务桥接组件，避免页面直接耦合未来真实组件路径
- 提供最小化的占位页面或演示入口，用于验证桥接结构可被业务层消费

## 非目标

- 不下载、拷贝或伪造真实 `uni-ui x` 组件源码
- 不实现业务页面
- 不将当前工程强行改造成 npm 工程
- 不假设所有 `uni-ui x` 组件都已可用

## 设计原则

### 1. 真实组件与业务代码解耦

业务页面不直接依赖 `uni_modules/uni-ui-x/**` 之类未来可能变化的真实插件路径，而是优先依赖仓库内桥接层。这样在真实组件导入后，只需要调整桥接层，不需要全局修改业务页面。

### 2. 当前可运行、未来可替换

在真实组件尚未导入前，桥接层提供占位实现，保证工程结构完整、页面可表达接入状态；在真实组件导入后，占位实现可平滑替换为真实实现。

### 3. 明确“已接入”和“待接入”状态

通过文档和组件命名显式区分占位实现与真实插件，避免团队误以为仓库已经完成真实接入。

## 方案拆分

### A. 文档层

新增接入说明文档，记录：

- 为什么当前仓库不直接使用 npm
- 后续在 HBuilderX 中手动导入 `uni-ui x` 的推荐路径
- 真实导入后需要替换的桥接点

### B. 目录层

新增以下目录约定：

- `components/ui/`
  - 项目对外使用的基础 UI 组件桥接层
- `components/ui/internal/`
  - 当前阶段的占位实现
- `docs/integration/`
  - 接入说明与操作手册

如果未来导入真实 `uni-ui x`，则由 HBuilderX 在根目录生成 `uni_modules/`，本次不手动伪造该目录内容。

### C. 组件层

先提供一个最小桥接组件，例如：

- `UiNavBar`
- `UiSectionCard`
- `UiEmptyState`

这些组件在当前阶段使用基础 `view`、`text`、`slot` 实现，占位表达“未来可切换到 uni-ui x”。后续真实导入 `uni-ui x` 后，可优先将 `UiNavBar` 对接 `uni-nav-bar`，其余继续保持自研或按实际需要替换。

### D. 演示层

新增一个非业务性质的接入准备页，用于说明：

- 当前项目尚未完成真实 `uni-ui x` 接入
- 当前桥接层已经建立
- 后续真实导入后的推荐替换点

## 推荐接入路径

1. 由开发者在 HBuilderX 中从插件市场导入 `uni-ui x` 套装或所需单组件
2. HBuilderX 在根目录生成真实 `uni_modules/`
3. 按兼容性优先选择真实组件，例如优先对接 `uni-nav-bar`
4. 修改仓库内桥接组件，使其内部切换到真实组件实现
5. 业务页面继续只引用桥接组件，不直接引用插件源码路径

## 风险与应对

### 风险 1：团队误将占位实现视为真实接入

应对：在页面文案、文档、组件注释中明确标注“占位桥接”。

### 风险 2：未来真实组件接口与当前假设不完全一致

应对：桥接层只暴露最小接口，避免过早绑定具体属性设计。

### 风险 3：当前仓库无法执行 `npm run lint`

应对：在交付说明中明确该阻塞点来自工程缺少 `package.json`，不额外伪造 npm 脚本。
